## 배경 및 목표

이 포트폴리오는 6년 전 프론트엔드 커리어 시작 시점에 처음으로 만든 웹사이트로, 당시의 학습 코드가 고스란히 남아 있다. 그 사이 프론트엔드 생태계는 크게 변화했고, 현재 코드베이스는 EOL(수명 종료)된 도구와 비표준 패턴으로 가득 차 있어 유지보수가 어렵다. 이 프로젝트는 사이트의 외형과 콘텐츠는 최대한 보존하면서, 코드베이스를 현대적이고 안정적인 기반 위에 재건하는 것을 목표로 한다.

- Q: 시각적 충실도(Fidelity) 기준 — 어느 수준까지 동일해야 하는가? → A: 기능적 동일. 색상·폰트·레이아웃 구성이 같으면 CSS 구현 방식에 따른 미세한 오차(1-2px)는 허용한다. 방문자가 인지할 수 있는 차이가 없으면 충분하다.

## 현재 상태 분석

### 주요 문제점

| 항목 | 현재 상태 | 심각도 |
|------|-----------|--------|
| 빌드 도구 | 없음 (Babel CLI만 존재) | 높음 |
| Babel 버전 | v6 (2017년부터 EOL) | 높음 |
| JSX 변환 | 브라우저에서 실시간 변환 (성능 저하) | 높음 |
| React 로딩 | CDN 스크립트 태그 방식 | 높음 |
| 타입 안전성 | 없음 | 중간 |
| 린팅/포매팅 | 없음 | 중간 |
| 테스트 | 없음 | 중간 |
| 접근성 | 부분적 (일부 이미지 alt 누락) | 중간 |
| 불필요한 의존성 | Lodash (Map만 사용) | 낮음 |
| 오타 (CSS 변수) | `--defult-font`, `--defult-height` | 낮음 |

### 현재 코드베이스 구조

```
src/
├── app.jsx          # App 컴포넌트, Container, Card
├── components.jsx   # 모든 UI 컴포넌트 (Experiences, Educations, Researches 등)
└── data.jsx         # 하드코딩된 포트폴리오 데이터
style/
├── style.css        # 메인 스타일
└── style.mobile.css # 모바일 스타일
```

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 빠르고 안정적인 포트폴리오 접근 (Priority: P1)

포트폴리오를 방문하는 채용 담당자나 동료 개발자가 사이트를 열었을 때, 불필요한 로딩 없이 즉시 콘텐츠를 확인할 수 있어야 한다. 현재는 브라우저가 직접 JSX를 컴파일하는 방식이라 초기 로딩이 느리며, CDN 의존으로 외부 장애 시 사이트 전체가 동작하지 않을 위험이 있다.

**Why this priority**: 포트폴리오 방문자의 첫인상을 결정하는 핵심 요소다. 느린 로딩은 인터뷰 기회를 잃는 직접적 원인이 될 수 있다.

**Independent Test**: 빌드된 사이트를 브라우저에서 열어 콘텐츠가 즉시 표시되는지 확인. Lighthouse 점수로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 방문자가 포트폴리오 URL에 접속할 때, **When** 페이지가 로드되면, **Then** 3초 이내에 소개·경험·학력·연구 모든 섹션이 표시된다.
2. **Given** 방문자가 모바일 기기로 접속할 때, **When** 페이지를 스크롤하면, **Then** 레이아웃이 화면 크기에 맞게 올바르게 표시된다.
3. **Given** 방문자가 네비게이션 링크를 클릭할 때, **When** 각 섹션 링크를 누르면, **Then** 해당 섹션으로 부드럽게 이동한다.
4. **Given** 방문자가 연구 항목의 PDF/링크를 클릭할 때, **When** 각 자료 버튼을 누르면, **Then** 해당 외부 링크나 파일이 올바르게 열린다.
5. **Given** CDN 서비스에 장애가 발생했을 때, **When** 방문자가 사이트에 접속하면, **Then** 외부 CDN 없이도 사이트가 정상 동작한다.

### User Story 2 - 콘텐츠 업데이트 (Priority: P2)

포트폴리오 소유자(나)가 새로운 경험, 학력, 연구 실적이 생겼을 때 빠르게 콘텐츠를 추가하거나 수정할 수 있어야 한다. 현재는 데이터가 컴포넌트 코드와 같은 파일에 섞여 있어 수정 시 컴포넌트 로직을 실수로 건드릴 위험이 있다.

**Why this priority**: 포트폴리오의 핵심 가치는 최신 정보를 담는 것이다. 현재 데이터가 코드와 혼재되어 수정 시 위험 부담이 크다.

**Independent Test**: 데이터 파일에 가상의 새 경험 항목을 추가하고 빌드 후 화면에 정상 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 포트폴리오 소유자가 새 직장 경험을 추가하려 할 때, **When** 데이터 파일에 새 항목을 입력하면, **Then** 컴포넌트 코드를 수정하지 않아도 화면에 반영된다.
2. **Given** 포트폴리오 소유자가 연구 논문 정보를 수정하려 할 때, **When** 데이터 파일의 해당 항목을 수정하면, **Then** 빌드 후 변경 사항이 정확히 반영된다.
3. **Given** 포트폴리오 소유자가 잘못된 데이터 형식으로 항목을 입력했을 때, **When** 빌드하면, **Then** 명확한 오류 메시지로 어떤 필드가 잘못되었는지 알 수 있다.

---

### User Story 3 - 코드 유지보수성 (Priority: P3)

포트폴리오 소유자(개발자로서)가 미래에 디자인을 수정하거나 새 섹션을 추가하려 할 때, 현대적인 도구의 지원을 받으며 안전하게 작업할 수 있어야 한다. 현재는 타입 안전성, 린터, 자동 포매팅이 없어 코드 품질이 개인의 주의에만 의존한다.

**Why this priority**: 포트폴리오는 장기적으로 관리되어야 하며, 도구의 도움 없이 코드를 수정하는 것은 실수를 유발하고 동기를 저하시킨다.

**Independent Test**: 타입 오류가 있는 코드를 의도적으로 작성하고 에디터 및 빌드 시 감지되는지 확인.

**Acceptance Scenarios**:

1. **Given** 개발자가 코드를 수정할 때, **When** 타입 오류가 있는 코드를 작성하면, **Then** 에디터(IDE)에서 즉시 오류를 표시한다.
2. **Given** 개발자가 코드를 저장할 때, **When** 코드 스타일이 기준에 맞지 않으면, **Then** 자동으로 포맷팅되거나 명확한 린팅 오류를 표시한다.
3. **Given** 개발자가 새 컴포넌트를 추가할 때, **When** 기존 패턴을 따라 작성하면, **Then** 코드 구조가 일관성 있게 유지된다.
4. **Given** 개발자가 GitHub에 코드를 푸시할 때, **When** 자동화된 빌드 프로세스가 실행되면, **Then** 별도의 수동 빌드 명령 없이 최신 사이트가 배포된다.

---

### User Story 4 - 접근성 및 SEO (Priority: P4)

스크린리더 등 보조 기술을 사용하는 방문자나 검색엔진이 포트폴리오의 콘텐츠에 올바르게 접근할 수 있어야 한다.

**Why this priority**: 포트폴리오는 가능한 많은 사람에게 노출되어야 하며, 접근성은 기본 품질 기준이다.

**Independent Test**: Lighthouse 접근성 검사 또는 스크린리더로 사이트를 탐색하여 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** 스크린리더 사용자가 사이트를 방문할 때, **When** 이미지 위에 포커스가 가면, **Then** 의미 있는 대체 텍스트가 읽힌다.
2. **Given** 검색엔진이 사이트를 크롤링할 때, **When** 페이지 메타데이터를 읽으면, **Then** 적절한 제목과 설명을 확인할 수 있다.
3. **Given** 키보드만으로 사이트를 탐색할 때, **When** Tab 키로 이동하면, **Then** 모든 대화형 요소(링크, 버튼)에 논리적인 순서로 접근 가능하다.

---

### Edge Cases

- 연구 자료(PDF, 포스터, 영상) 링크가 깨진 경우, 해당 버튼은 적절한 안내를 표시한다.
- 이미지 파일(프로필 사진, 연구 썸네일)이 느린 네트워크에서 로드되지 않을 때 레이아웃이 깨지지 않는다.
- 콘텐츠 데이터에 필수 필드가 누락된 경우 빌드 단계에서 오류로 감지한다.
- 매우 긴 텍스트(저자 목록, 논문 제목)가 입력되어도 레이아웃이 깨지지 않는다.

## Requirements *(mandatory)*

### Functional Requirements

**빌드 및 배포**

- **FR-001**: 시스템은 소스 코드로부터 최적화된 정적 파일을 빌드하는 자동화된 파이프라인을 제공해야 한다.
- **FR-002**: 빌드된 결과물은 브라우저에서 직접 JSX를 변환하지 않아야 한다.
- **FR-003**: GitHub Pages에 코드를 푸시하면 자동으로 최신 버전이 배포되어야 한다.
- **FR-004**: 개발 환경에서 소스 코드 변경 시 브라우저가 자동으로 갱신되어야 한다.
- **FR-005**: 외부 CDN 없이 모든 의존 라이브러리가 빌드 결과물에 포함되어야 한다.

**코드 품질**

- **FR-006**: 모든 소스 코드는 타입 정의를 포함하며, 타입 오류는 빌드 단계에서 감지되어야 한다.
- **FR-007**: 코드 린팅 규칙 위반이 빌드 시 또는 에디터에서 실시간으로 감지되어야 한다.
- **FR-008**: 코드 포맷팅은 자동화된 규칙에 의해 일관성 있게 유지되어야 한다.
- **FR-009**: 현재 사용하지 않는 외부 라이브러리 의존성을 제거해야 한다.

**콘텐츠 관리**

- **FR-010**: 포트폴리오 데이터(경험, 학력, 연구)는 컴포넌트 코드와 분리되어 독립적으로 편집 가능해야 한다.
- **FR-011**: 데이터 구조는 타입으로 정의되어, 잘못된 형식의 데이터를 빌드 시 감지해야 한다.
- **FR-012**: 각 포트폴리오 섹션(경험, 학력, 연구)은 데이터 파일만 수정하여 콘텐츠를 추가·수정할 수 있어야 한다.

**디자인·콘텐츠 보존 (Fidelity)**

- **FR-013**: 현재 포트폴리오의 모든 섹션(소개, 경험, 학력, 연구)의 시각적 출력이 기존과 기능적으로 동일하게 유지되어야 한다. 색상, 레이아웃, 폰트, 섹션 구성 등 방문자가 인지할 수 있는 시각 요소가 보존 대상이며, CSS 구현 방식 차이로 인한 미세한 오차(1-2px)는 허용한다.
- **FR-014**: 현재의 단일 페이지 스크롤 내비게이션 방식이 유지되어야 한다.
- **FR-015**: 모바일 반응형 레이아웃이 유지되어야 한다.
- **FR-016**: CV PDF 다운로드 및 외부 링크(LinkedIn 등)가 정상 동작해야 한다.
- **FR-020**: 접근성·SEO·유지보수성 향상을 위해 불가피하게 HTML 구조나 콘텐츠를 변경하는 경우, 해당 변경 사항은 PR에서 변경 이유와 영향 범위를 명시하여 검토받아야 한다.

**접근성**

- **FR-017**: 모든 이미지에는 의미 있는 대체 텍스트(alt)가 제공되어야 한다.
- **FR-018**: 모든 대화형 요소는 키보드로 접근 가능해야 한다.
- **FR-019**: 페이지는 적절한 HTML 시맨틱 구조(heading 계층, landmark)를 가져야 한다.

### Key Entities

- **Experience(경험)**: 직책, 회사명, 재직 기간, 역할 설명, 외부 링크
- **Education(학력)**: 학교명, 학위, 전공, 재학 기간, 지도교수(선택)
- **Research(연구)**: 논문 제목, 저자 목록, 발표 정보(학회/저널명, 연도), 자료 링크(PDF, 포스터, 영상), 썸네일 이미지
- **Material(자료)**: 자료 유형(PDF/포스터/영상/슬라이드), URL, 표시 여부

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 페이지 초기 로딩 시 주요 콘텐츠가 3초 이내에 표시된다 (브라우저 내 컴파일 제거로 인한 성능 개선).
- **SC-002**: Lighthouse 성능 점수 90점 이상, 접근성 점수 90점 이상을 달성한다.
- **SC-003**: 새로운 경험·학력·연구 항목을 데이터 파일에 추가하고 빌드까지 5분 이내에 완료할 수 있다.
- **SC-004**: 코드 작성 시 타입 오류와 린팅 위반을 에디터에서 실시간으로 확인할 수 있다.
- **SC-005**: 현재 포트폴리오의 모든 콘텐츠(경험, 학력, 연구 등)가 새 코드베이스에서 동일하게 표시된다.
- **SC-006**: GitHub에 코드 푸시 후 10분 이내에 최신 버전이 GitHub Pages에 반영된다.
- **SC-007**: 외부 CDN 의존성이 빌드 결과물에 내재화되어, CDN 장애 시에도 사이트가 정상 동작한다.

## Assumptions

이 스펙은 다음 사항을 가정한다:

1. **단일 페이지 유지**: 포트폴리오는 라우팅이 필요 없는 단일 스크롤 페이지로 유지한다.
2. **정적 배포 유지**: GitHub Pages에 정적 파일을 배포하는 현재 방식을 유지한다.
3. **디자인·콘텐츠 보존**: 시각적 디자인(색상, 레이아웃, 폰트, 섹션 구성)과 모든 텍스트·미디어 콘텐츠는 현재와 동일하게 유지한다. 내부 코드·마크업 구조는 변경될 수 있으나, 방문자가 보고 경험하는 결과물은 바뀌지 않아야 한다. **예외**: 접근성·SEO·유지보수성에 중대한 영향을 끼치는 경우 제한적 수정을 허용하되, 반드시 PR에서 변경 이유·영향 범위를 명시하고 검토받는다.
4. **콘텐츠 보존**: 현재의 경력, 학력, 연구 데이터를 그대로 이전한다. 데이터 구조(파일 분리, 타입 정의)는 변경될 수 있으나 실제 콘텐츠 값(이름, 날짜, 설명문 등)은 수정 없이 이전한다.
5. **언어**: 현재의 한/영 혼용 콘텐츠 구조를 유지한다. 다국어 지원 시스템은 불필요하다.
6. **에셋 보존**: `assets/` 폴더의 PDF, 이미지, 영상 파일은 그대로 유지한다.
7. **보수적 기술 선정**: 실험적이거나 최신 베타 버전의 도구보다 안정적이고 커뮤니티에서 검증된 도구를 우선한다.
8. **CSS 아키텍처**: 어떤 CSS 방식을 사용할지는 planning 단계에서 결정한다. 시각적 결과가 기존과 동일하게 유지되는 한 vanilla CSS·CSS Modules·Tailwind CSS 등 어느 방식이든 허용된다.

## Out of Scope

- 포트폴리오 디자인 변경 또는 새로운 UI 섹션 추가
- CMS(콘텐츠 관리 시스템) 연동
- 다국어(i18n) 지원 체계 구축
- 서버사이드 렌더링(SSR) 또는 백엔드 API
- 기존 자산 파일(PDF, 이미지) 최적화 또는 압축
- 새로운 포트폴리오 콘텐츠 추가 (코드 이전 완료 후 별도 작업)
- 단위 테스트 또는 E2E 테스트 작성 (향후 개선 사항)
